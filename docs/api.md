# เอกสาร API

เว็บไซต์นี้ไม่ได้เขียนเซิร์ฟเวอร์ API ขึ้นมาเองทั้งหมด แต่ใช้ API ที่ Supabase สร้างให้อัตโนมัติจากโครงสร้างฐานข้อมูล (เรียกว่า REST API ผ่าน PostgREST) หน้าเว็บเรียกใช้ API เหล่านี้โดยตรงผ่านไลบรารี `@supabase/supabase-js`

เอกสารนี้อธิบายว่า API ที่ใช้งานจริงมีอะไรบ้าง และมี API อีกชุดหนึ่งที่เตรียมไว้แต่ยังไม่ได้ใช้งานจริง (ต้องแยกให้ชัดเพื่อไม่ให้สับสน)

## API ที่ใช้งานจริงในเว็บไซต์ (ผ่าน Supabase)

รูปแบบ URL คือ `https://<โปรเจกต์ของคุณ>.supabase.co/rest/v1/<ชื่อตาราง>` ทุกคำขอต้องแนบกุญแจ (API key) แบบ publishable/anon key ที่ตั้งไว้ในไฟล์ `.env` ของโปรเจกต์ ระบบจะตรวจสิทธิ์ตามกฎ RLS ที่อธิบายไว้ใน [data-schema.md](./data-schema.md) เสมอ ไม่ว่าจะเรียกจากที่ไหน

| การทำงาน | คำขอ | ใครเรียกได้ |
| --- | --- | --- |
| ดูรายการสินค้าทั้งหมด | `GET /rest/v1/products?select=*&order=id` | ทุกคน |
| เพิ่มสินค้าใหม่ | `POST /rest/v1/products` | เฉพาะแอดมินที่เข้าสู่ระบบแล้ว |
| แก้ไขสินค้า | `PATCH /rest/v1/products?id=eq.<รหัสสินค้า>` | เฉพาะแอดมินที่เข้าสู่ระบบแล้ว |
| ลบสินค้า | `DELETE /rest/v1/products?id=eq.<รหัสสินค้า>` | เฉพาะแอดมินที่เข้าสู่ระบบแล้ว |
| สมัครสมาชิก | `POST /auth/v1/signup` | ทุกคน |
| เข้าสู่ระบบ | `POST /auth/v1/token?grant_type=password` | ทุกคน |
| ดู/แก้ไขข้อมูลโปรไฟล์ตัวเอง | `GET` / `PATCH` `/rest/v1/profiles?id=eq.<รหัสผู้ใช้>` | เจ้าของบัญชีเท่านั้น |

สเปกแบบละเอียดของ API ชุดนี้อยู่ในไฟล์ [swagger.yaml](./swagger.yaml) สามารถเปิดดูแบบมีหน้าตาสวยงามได้โดยนำไฟล์นี้ไปวางใน [Swagger Editor](https://editor.swagger.io) หรือเครื่องมือที่รองรับ OpenAPI

## ระบบหลังบ้านสำรอง (Cloudflare Worker) ที่ยังไม่ได้ใช้งานจริง

ในโฟลเดอร์ `backend/src/index.ts` มีโค้ดเซิร์ฟเวอร์ที่เขียนไว้ล่วงหน้า มี 2 endpoint ดังนี้

| Endpoint | คำอธิบาย | สถานะ |
| --- | --- | --- |
| `GET /api/products` | ส่งข้อมูลสินค้าตัวอย่างที่ใส่ไว้ตายตัวในโค้ด (ไม่ได้ดึงจากฐานข้อมูลจริง) | ตัวอย่าง (mock) ไม่ได้ใช้งานจริงในเว็บ |
| `POST /api/payments/verify` | จำลองการตรวจสอบยอดโอนเงิน โดยเช็คว่ายอดตรงกับ 15900 บาทหรือไม่ | ตัวอย่าง (mock) ไม่ได้ใช้งานจริงในเว็บ |

หน้าเว็บที่ใช้งานจริงไม่ได้เรียก endpoint เหล่านี้เลยแม้แต่ครั้งเดียว เก็บไว้เป็นแนวทางสำหรับพัฒนาต่อในอนาคต เช่น ถ้าต้องการทำระบบตรวจสลิปโอนเงินจริงกับผู้ให้บริการอย่าง EasySlip ควรพัฒนาต่อจากจุดนี้

## ทำไมถึงไม่รวมทั้งสองส่วนเข้าด้วยกัน

การให้หน้าเว็บคุยกับ Supabase โดยตรงทำให้ระบบเรียบง่ายและพัฒนาไว เหมาะกับฟีเจอร์ที่มีอยู่ตอนนี้ ส่วน Cloudflare Worker จะมีประโยชน์เมื่อต้องทำงานที่ Supabase ทำเองไม่ได้ เช่น เชื่อมต่อกับระบบภายนอกที่ต้องเก็บรหัสลับไว้ในเซิร์ฟเวอร์ (ไม่ควรเก็บไว้ในหน้าเว็บที่ผู้ใช้มองเห็นโค้ดได้) เมื่อถึงเวลานั้นค่อยเชื่อมสองส่วนเข้าด้วยกัน
