# รายงานผลการทดสอบ

เอกสารนี้สรุปผลการทดสอบระบบก่อนนำเสนอลูกค้า ทดสอบล่าสุดเมื่อวันที่ 22 กรกฎาคม 2569 โดยจำลองการใช้งานของผู้เยี่ยมชม สมาชิก และแอดมิน ผ่านเบราว์เซอร์อัตโนมัติจริง (ไม่ใช่แค่อ่านโค้ด) ครบทั้ง 14 รายการตามเอกสาร [uat.md](./uat.md) พร้อมตรวจสอบข้อมูลในฐานข้อมูลจริงประกอบ

## สรุปผลโดยรวม

ทุกรายการที่ทดสอบผ่านตามที่คาดหวังครบทั้ง 14 ข้อ **ระบบพร้อมให้เจ้าของร้าน/ลูกค้าทำแบบทดสอบการยอมรับระบบ (UAT) ด้วยตัวเองตามเอกสาร [uat.md](./uat.md) เพื่อลงนามยอมรับต่อไปได้**

## วิธีการทดสอบ

ทดสอบ 3 ระดับควบคู่กัน

1. **ทดสอบผ่านเบราว์เซอร์จริง**: เปิดเว็บไซต์จริง (รันเซิร์ฟเวอร์พัฒนาขึ้นมาทดสอบ) แล้วสั่งงานผ่านเบราว์เซอร์อัตโนมัติ (Playwright ควบคุม Chromium) ให้ทำตามทุกขั้นตอนในเอกสาร UAT ทีละข้อ ตั้งแต่ผู้เยี่ยมชม สมัครสมาชิก จนถึงแอดมินจัดการสินค้า พร้อมเก็บภาพหน้าจอทุกขั้นตอนไว้เป็นหลักฐาน
2. **ทดสอบข้อมูลในฐานข้อมูลโดยตรง**: ตรวจสอบผลลัพธ์แต่ละขั้นตอนกับฐานข้อมูลจริงคู่กัน (เช่น เช็คว่าสมาชิกใหม่ถูกสร้างในตาราง `profiles` จริง สินค้าทดสอบถูกลบออกจริงไม่มีตกค้าง) และเก็บกวาดบัญชี/ข้อมูลที่สร้างขึ้นเพื่อการทดสอบออกจากระบบหลังทดสอบเสร็จ ไม่ทิ้งข้อมูลปลอมไว้ในระบบจริง
3. **ตรวจสอบความปลอดภัยของฐานข้อมูล**: รันตัวตรวจสอบความปลอดภัยของ Supabase (Security Advisor) เพื่อยืนยันว่าไม่มีช่องโหว่ใหม่เกิดขึ้น

## ผลการทดสอบรายข้อ (ตรงกับลำดับในเอกสาร uat.md)

| ข้อ | รายการทดสอบ | ผลจริง |
| --- | --- | --- |
| 1 | เปิดหน้าแรกโดยไม่เข้าสู่ระบบ เห็นสินค้าพร้อมรูป ชื่อ ราคา | ผ่าน เห็นสินค้าครบ 4 รายการ |
| 2 | กด "หยิบใส่ตะกร้า" ตัวเลขที่ไอคอนตะกร้าเพิ่มขึ้น | ผ่าน ตัวเลขเปลี่ยนจากไม่มีเป็น 1 |
| 3 | ไปหน้าตะกร้า เห็นสินค้าที่เพิ่งใส่ | ผ่าน |
| 4 | พิมพ์ /admin เอง (ยังไม่เข้าสู่ระบบ) เห็นข้อความสงวนสิทธิ์ | ผ่าน แสดง "สงวนสิทธิ์เฉพาะผู้ดูแลระบบ" ไม่มีข้อมูลรั่วไหล |
| 5 | สมัครสมาชิกใหม่ผ่านฟอร์ม | ผ่าน ขึ้นข้อความ "สมัครสมาชิกสำเร็จ คุณสามารถเข้าสู่ระบบได้เลย" (ระบบไม่บังคับยืนยันอีเมลก่อนใช้งาน) |
| 6 | เข้าสู่ระบบด้วยบัญชีที่เพิ่งสมัคร | ผ่าน เห็นชื่อผู้ใช้ที่มุมขวาบนทันที |
| 7 | สมาชิกทั่วไปพิมพ์ /admin เอง | ผ่าน แสดง "ไม่มีสิทธิ์เข้าถึง" ตรวจสอบกับฐานข้อมูลแล้วบัญชีนี้มี role เป็น customer จริง |
| 8 | เข้าสู่ระบบด้วยบัญชีแอดมิน เปิดหน้า /admin | ผ่าน เห็นแดชบอร์ดครบ |
| 9 | ตัวเลข "สมาชิกรวม" ในแดชบอร์ด | ผ่าน แสดง 4 คน ตรงกับจำนวนจริงในฐานข้อมูล ไม่ใช่ 0 |
| 10 | เพิ่มสินค้าใหม่ กรอกชื่อ ราคา รายละเอียด รูปภาพ แล้วบันทึก | ผ่าน ขึ้นข้อความ "เพิ่มสินค้าใหม่เรียบร้อย" และปรากฏในรายการทันทีโดยไม่ต้องรีเฟรชหน้า |
| 11 | แก้ไขชื่อสินค้าที่เพิ่งเพิ่ม แล้วบันทึก | ผ่าน ขึ้นข้อความ "อัปเดตข้อมูลสินค้าเรียบร้อย" ชื่อใหม่แสดงถูกต้องทั้งในรายการและช่องแก้ไข |
| 12 | กดลบสินค้าทดสอบ | ผ่าน ระบบถามยืนยันก่อนลบจริง กดยืนยันแล้วสินค้าหายจากรายการ และตรวจสอบกับฐานข้อมูลแล้วไม่มีแถวตกค้าง |
| 13 | เปิดหน้าแรกอีกครั้งหลังแก้ไขสินค้าในหน้าแอดมิน | ผ่าน หน้าแรกไม่มีสินค้าทดสอบหลงเหลือ ข้อมูลตรงกับที่แก้ไขในหน้าแอดมิน |
| 14 | ออกจากระบบแล้วเข้าสู่ระบบใหม่ด้วยบัญชีแอดมินเดิม | ผ่าน เข้าหน้าแอดมินได้เหมือนเดิม สิทธิ์ไม่หายไปหลังออกจากระบบ |

## ข้อสังเกตทางเทคนิคที่ควรทราบ

- บัญชีที่ใช้ทดสอบข้อ 5-14 เป็นบัญชีที่สร้างขึ้นเฉพาะสำหรับการทดสอบรอบนี้ (ตั้ง role เป็น admin ผ่านฐานข้อมูลโดยตรงเพื่อทดสอบสิทธิ์แอดมิน โดยไม่ได้แก้ไขบัญชีแอดมินจริงที่มีอยู่) และลบออกจากระบบเรียบร้อยหลังทดสอบเสร็จ ไม่มีข้อมูลทดสอบตกค้างในระบบจริง
- ระบบสมัครสมาชิกไม่ได้บังคับให้ยืนยันอีเมลก่อนเข้าสู่ระบบ (สมัครแล้วเข้าใช้งานได้ทันที) หากต้องการความปลอดภัยเพิ่มขึ้นควรพิจารณาเปิดการยืนยันอีเมลในการตั้งค่า Supabase Auth ในรอบถัดไป
- ตรวจสอบ Security Advisor ของฐานข้อมูลแล้ว ไม่พบช่องโหว่ระดับสูงใหม่ ปัญหาที่มีอยู่ (ตาราง `profiles_admin` เปิดสิทธิ์กว้างเกินไป, คอลัมน์ `password` ที่ไม่ได้ใช้งานในตาราง `profiles`, ยังไม่เปิด Leaked Password Protection) เป็นรายการเดิมที่บันทึกไว้แล้วใน [analysis-and-design.md](./analysis-and-design.md) และไม่กระทบต่อฟีเจอร์ที่ทดสอบในรอบนี้

## สิ่งที่ไม่ได้อยู่ในขอบเขตการทดสอบรอบนี้

การทดสอบรอบนี้เน้นเฉพาะเรื่องการแสดงสินค้า ตะกร้า สมาชิก และการจัดการสินค้าในหน้าแอดมิน ไม่ได้ครอบคลุมเรื่องต่อไปนี้ เพราะยังไม่ได้พัฒนา (ดูรายละเอียดใน [analysis-and-design.md](./analysis-and-design.md))

- การยืนยันคำสั่งซื้อและชำระเงินจริง
- การตรวจสอบสลิปโอนเงินอัตโนมัติ
- หน้าประวัติคำสั่งซื้อของลูกค้า
- การแก้ไขข้อมูลโปรไฟล์ส่วนตัว (พบว่ามีบั๊กอยู่ระหว่างตรวจสอบ บันทึกไว้ในเอกสารการวิเคราะห์และออกแบบแล้ว)

## ขั้นตอนต่อไป

ระบบผ่านการทดสอบภายในครบทุกข้อแล้ว ขั้นตอนต่อไปคือให้เจ้าของร้าน (ลูกค้า) ทำแบบทดสอบด้วยตัวเองตามเอกสาร [uat.md](./uat.md) เพื่อยืนยันและลงนามยอมรับระบบก่อนนำไปใช้งานจริง
