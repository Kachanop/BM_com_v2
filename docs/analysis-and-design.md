# การวิเคราะห์และออกแบบระบบ

เอกสารนี้สรุปว่าทำไมระบบถึงถูกออกแบบมาแบบนี้ อะไรคือความต้องการหลัก และมีเรื่องอะไรที่ควรพัฒนาต่อในอนาคต

## ความต้องการของระบบ (Requirements)

### ความต้องการด้านการทำงาน (Functional Requirements)

| รหัส | ความต้องการ | สถานะ |
| --- | --- | --- |
| FR-1 | ลูกค้าดูรายการสินค้าได้โดยไม่ต้องเข้าสู่ระบบ | ใช้งานได้แล้ว |
| FR-2 | ลูกค้าใส่สินค้าลงตะกร้าและปรับจำนวนได้ | ใช้งานได้แล้ว |
| FR-3 | ผู้ใช้สมัครสมาชิกและเข้าสู่ระบบด้วยอีเมล/รหัสผ่านได้ | ใช้งานได้แล้ว |
| FR-4 | แอดมินเพิ่ม แก้ไข และลบข้อมูลสินค้าได้ | ใช้งานได้แล้ว (แก้ไขในรอบงานนี้) |
| FR-5 | แอดมินดูสรุปยอดขาย จำนวนคำสั่งซื้อ และจำนวนสมาชิกได้ | ใช้งานได้แล้ว (ตัวเลขจะเป็น 0 จนกว่าจะมีคำสั่งซื้อจริง) |
| FR-6 | ลูกค้ายืนยันสั่งซื้อและชำระเงินผ่าน QR พร้อมเพย์ | ยังไม่ได้พัฒนา |
| FR-7 | ระบบตรวจสอบสลิปโอนเงินอัตโนมัติ | ยังไม่ได้พัฒนา |
| FR-8 | ลูกค้าดูประวัติคำสั่งซื้อของตัวเอง | ยังไม่ได้พัฒนา (หน้าเปล่ารอการเชื่อมข้อมูลจริง) |
| FR-9 | ผู้ใช้แก้ไขข้อมูลส่วนตัว (ที่อยู่ เบอร์โทร ช่องทางชำระเงิน) | มีหน้าจอแล้วแต่มีบั๊ก (ดูหัวข้อปัญหาที่พบ) |

### ความต้องการด้านคุณภาพ (Non-Functional Requirements)

- **ความปลอดภัยของข้อมูล**: ต้องมีการควบคุมสิทธิ์ในระดับฐานข้อมูล ไม่ใช่แค่ในหน้าเว็บ เพื่อป้องกันกรณีมีคนพยายามเรียก API ตรงโดยข้ามหน้าเว็บ
- **ใช้งานง่าย**: หน้าจอสำหรับแอดมินต้องเข้าใจง่าย ไม่ต้องมีความรู้ด้านเทคนิคก็ใช้งานได้
- **ความถูกต้องของข้อมูล**: ข้อมูลสินค้าที่แอดมินแก้ไขต้องสะท้อนไปยังหน้าลูกค้าทันทีในการโหลดครั้งถัดไป

## ผู้ใช้งานระบบ (Actors)

ดูรายละเอียดใน [use-case-and-sequence.md](./use-case-and-sequence.md) สรุปสั้นคือมี 3 บทบาท ได้แก่ ผู้เยี่ยมชม ลูกค้าสมาชิก และแอดมิน

## การตัดสินใจด้านการออกแบบที่สำคัญ

### เลือกให้หน้าเว็บคุยกับฐานข้อมูลโดยตรง แทนที่จะผ่านเซิร์ฟเวอร์กลาง

เหตุผลคือ Supabase มีระบบควบคุมสิทธิ์ระดับแถวข้อมูล (Row Level Security) ในตัวอยู่แล้ว ทำให้ไม่จำเป็นต้องเขียนเซิร์ฟเวอร์กลางเพิ่มสำหรับงานพื้นฐานอย่างการอ่าน/เขียนข้อมูลสินค้า ลดความซับซ้อนของระบบและพัฒนาได้เร็วขึ้น รายละเอียดเพิ่มเติมอยู่ใน [tech-stack.md](./tech-stack.md)

### แก้ปัญหาสิทธิ์การเข้าถึงข้อมูลด้วยฟังก์ชันตรวจสอบสิทธิ์กลาง

ก่อนหน้านี้ระบบเปิด RLS ไว้ในทุกตารางแต่ไม่มีกฎอนุญาตเลย ทำให้อ่านข้อมูลไม่ได้ทั้งหมด การแก้ไขในรอบนี้สร้างฟังก์ชัน `is_admin()` ขึ้นมาเป็นจุดกลางสำหรับตรวจสอบว่าใครเป็นแอดมิน แทนที่จะเขียนเงื่อนไขซ้ำๆ ในทุกกฎ ทำให้ดูแลรักษาง่ายขึ้นในอนาคต รายละเอียดอยู่ใน [data-schema.md](./data-schema.md) และ [test-report.md](./test-report.md)

## ปัญหาที่พบระหว่างตรวจสอบระบบ (สำหรับพิจารณาแก้ไขในรอบถัดไป)

รายการนี้เป็นสิ่งที่พบระหว่างการตรวจสอบระบบในรอบนี้ แต่ไม่ได้อยู่ในขอบเขตงานที่ตกลงกันไว้ (ขอบเขตงานรอบนี้คือแก้บั๊กสินค้าหายและเพิ่มระบบจัดการสินค้า) จึงบันทึกไว้เพื่อพิจารณาแก้ไขในรอบถัดไป

| ปัญหา | รายละเอียด | ความรุนแรง |
| --- | --- | --- |
| หน้าแก้ไขโปรไฟล์บันทึกไม่ได้ | หน้าจอแก้ไขโปรไฟล์ (`ProfileEditModal.jsx`) พยายามบันทึกข้อมูลลงคอลัมน์ `address` และ `payment_method` แต่ตาราง `profiles` ในฐานข้อมูลจริงไม่มีสองคอลัมน์นี้อยู่ ทำให้การบันทึกจะล้มเหลวทุกครั้ง ต้องเพิ่มคอลัมน์ในฐานข้อมูล หรือปรับหน้าจอให้ตรงกับคอลัมน์ที่มีอยู่จริง | สูง (ฟีเจอร์ใช้งานไม่ได้เลย) |
| ระบบสั่งซื้อยังไม่เชื่อมกับฐานข้อมูลจริง | หน้าตะกร้าไม่มีปุ่มยืนยันสั่งซื้อที่บันทึกลงตาราง orders/order_items หน้าประวัติคำสั่งซื้อเป็นข้อความคงที่ | สูง (เป็นฟีเจอร์หลักที่ README เดิมเคยพูดถึง) |
| ตาราง profiles_admin ไม่ได้ใช้งานแต่เปิดสิทธิ์กว้างเกินไป | ไม่มีโค้ดฝั่งหน้าเว็บเรียกใช้ตารางนี้เลย แต่มีกฎ RLS แบบอนุญาตทุกการกระทำ ควรพิจารณาลบตารางทิ้งเพื่อลดความเสี่ยง | ปานกลาง (ไม่มีข้อมูลอยู่ในตอนนี้ แต่เป็นความเสี่ยงถ้ามีคนเผลอใส่ข้อมูลเข้าไป) |
| คอลัมน์ password ในตาราง profiles | มีคอลัมน์นี้อยู่แต่ไม่ได้ใช้งานจากโค้ดหน้าเว็บเลย (ระบบสมาชิกจริงใช้ Supabase Auth ซึ่งเก็บรหัสผ่านแยกต่างหากอย่างปลอดภัยอยู่แล้ว) ควรพิจารณาลบคอลัมน์นี้ทิ้งเพื่อไม่ให้เกิดความสับสนหรือความเสี่ยงในอนาคต | ต่ำ (ไม่ได้ใช้งาน แต่ควรทำความสะอาด) |
| ยังไม่มีการป้องกันรหัสผ่านรั่วไหล | Supabase มีฟีเจอร์ตรวจรหัสผ่านที่เคยรั่วไหลจากที่อื่น (Leaked Password Protection) แต่ยังไม่ได้เปิดใช้งานในโปรเจกต์นี้ | ต่ำ |

## แผนงานที่แนะนำสำหรับรอบพัฒนาถัดไป

1. แก้ไขหน้าแก้ไขโปรไฟล์ให้ตรงกับโครงสร้างฐานข้อมูลจริง
2. พัฒนาระบบยืนยันคำสั่งซื้อให้บันทึกลงตาราง orders/order_items จริง แล้วเชื่อมหน้าประวัติคำสั่งซื้อให้ดึงข้อมูลจริงมาแสดง
3. พัฒนาระบบสร้าง QR พร้อมเพย์และตรวจสลิปอัตโนมัติ (จุดนี้อาจเป็นจังหวะที่เหมาะสมในการเริ่มใช้งาน Cloudflare Worker ที่เตรียมไว้ เพราะการเรียกผู้ให้บริการภายนอกควรทำผ่านเซิร์ฟเวอร์ ไม่ควรทำจากหน้าเว็บโดยตรง)
4. พิจารณาลบตาราง profiles_admin และคอลัมน์ password ที่ไม่ได้ใช้งาน
