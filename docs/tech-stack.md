# เทคโนโลยีที่ใช้ในโปรเจกต์

เอกสารนี้อธิบายว่าเว็บไซต์บ้านมีคอม V2 สร้างขึ้นมาด้วยอะไรบ้าง และแต่ละส่วนทำหน้าที่อะไร เขียนให้อ่านเข้าใจง่าย ไม่ต้องมีพื้นฐานเขียนโปรแกรมมาก่อนก็อ่านตามได้

## ภาพรวมส่วนประกอบ

เว็บไซต์แบ่งเป็น 3 ส่วนหลัก เหมือนร้านค้าจริงที่มีหน้าร้าน หลังร้าน และห้องเก็บของ

1. **หน้าเว็บ (Frontend)** คือส่วนที่ลูกค้าและแอดมินมองเห็นและกดใช้งานผ่านเบราว์เซอร์
2. **ฐานข้อมูล (Database)** คือที่เก็บข้อมูลสินค้า คำสั่งซื้อ และสมาชิกทั้งหมด
3. **ระบบหลังบ้านสำรอง (Backend Worker)** เป็นโค้ดที่เตรียมไว้สำหรับอนาคต ตอนนี้ยังไม่ได้ใช้งานจริงในเว็บ (อธิบายเพิ่มในหัวข้อท้ายเอกสาร)

## ตารางเทคโนโลยี

| ส่วน | เทคโนโลยีที่ใช้ | ทำหน้าที่อะไร (พูดง่ายๆ) |
| --- | --- | --- |
| หน้าเว็บ | React 19 + Vite | เครื่องมือสร้างหน้าเว็บ ทำให้กดปุ่มแล้วหน้าจอเปลี่ยนทันทีโดยไม่ต้องโหลดหน้าใหม่ทั้งหมด |
| ความสวยงามของหน้าเว็บ | Tailwind CSS 4 | ชุดเครื่องมือแต่งหน้าเว็บ กำหนดสี ระยะห่าง และความสวยงามให้เหมือนกันทุกหน้า |
| การเปลี่ยนหน้า | React Router | ทำให้กดลิงก์แล้วเปลี่ยนหน้า (หน้าแรก ตะกร้า ประวัติคำสั่งซื้อ แอดมิน) โดยไม่ต้องโหลดใหม่ |
| ไอคอนต่างๆ | lucide-react | ชุดไอคอนสำเร็จรูปที่ใช้ตกแต่งปุ่มและเมนู |
| ฐานข้อมูลและระบบสมาชิก | Supabase (PostgreSQL) | เก็บข้อมูลสินค้า คำสั่งซื้อ และสมาชิกแบบเรียลไทม์ พร้อมระบบสมัคร/เข้าสู่ระบบในตัว |
| การเชื่อมต่อฐานข้อมูล | @supabase/supabase-js | ตัวช่วยให้หน้าเว็บคุยกับฐานข้อมูล Supabase ได้โดยตรงจากเบราว์เซอร์ |
| ระบบหลังบ้านสำรอง | Cloudflare Worker + Hono | โค้ดฝั่งเซิร์ฟเวอร์ที่เตรียมไว้ ยังไม่ถูกเรียกใช้งานจริงในตอนนี้ |
| ที่อยู่เว็บไซต์ (แผนงาน) | Cloudflare Pages | พื้นที่สำหรับวางเว็บไซต์ให้คนทั่วไปเข้าถึงได้ |
| เครื่องมือของทีมพัฒนา | Git + GitHub | ที่บันทึกการแก้ไขโค้ดของทีม ย้อนดูประวัติการแก้ไขได้ |
| ทดสอบระบบอัตโนมัติ | Playwright | E2E testing ครอบคลุม Homepage / Cart / Stock behavior รวม 23 test cases รันบน Chromium จำลองพฤติกรรมผู้ใช้จริง |

## ทำไมถึงเลือกใช้ Supabase โดยตรงจากหน้าเว็บ

ปกติเว็บไซต์หลายแห่งจะมีเซิร์ฟเวอร์ตรงกลางคอยรับคำขอจากหน้าเว็บ แล้วค่อยไปคุยกับฐานข้อมูลอีกที แต่ในโปรเจกต์นี้ หน้าเว็บ (React) คุยกับฐานข้อมูล Supabase โดยตรง เพราะ Supabase มีระบบตรวจสิทธิ์ในตัว (เรียกว่า Row Level Security หรือ RLS) ทำให้กำหนดได้ว่าใครอ่าน/แก้ไขข้อมูลอะไรได้บ้าง โดยไม่ต้องเขียนเซิร์ฟเวอร์กลางเพิ่มเอง เหมาะกับโปรเจกต์ขนาดนี้ที่ต้องการความเรียบง่ายและพัฒนาไว

## เรื่องของ Cloudflare Worker (ระบบหลังบ้านสำรอง)

ในโฟลเดอร์ `backend` มีโค้ดเซิร์ฟเวอร์ที่เขียนด้วย Cloudflare Worker และเฟรมเวิร์ก Hono เตรียมไว้ แต่ ณ ตอนนี้หน้าเว็บจริงไม่ได้เรียกใช้งานโค้ดส่วนนี้เลย ข้อมูลที่ส่งกลับมาจาก Worker เป็นข้อมูลตัวอย่าง (mock data) ที่ใส่ไว้ล่วงหน้าเท่านั้น ถ้าในอนาคตต้องการทำระบบที่ซับซ้อนขึ้น เช่น ตรวจสลิปโอนเงินอัตโนมัติ ค่อยพัฒนาโค้ดส่วนนี้ต่อและเชื่อมกับหน้าเว็บจริงอีกที รายละเอียดเพิ่มเติมอยู่ใน [api.md](./api.md)
