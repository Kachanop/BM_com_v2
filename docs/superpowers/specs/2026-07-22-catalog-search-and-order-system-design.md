# สเปกออกแบบ: ระบบค้นหา/หมวดหมู่สินค้า, หน้ารายละเอียดสินค้า, ยืนยันสั่งซื้อ, ใบเสร็จ, ประวัติคำสั่งซื้อ และจัดการคำสั่งซื้อฝั่งแอดมิน

วันที่: 2026-07-22

> เอกสารนี้รวมและแทนที่ [2026-07-22-checkout-order-history-design.md](./2026-07-22-checkout-order-history-design.md) เดิม (เนื้อหาส่วนคำสั่งซื้อ/ชำระเงิน/ประวัติ/แอดมินยังเหมือนเดิมทุกประการ) และเพิ่มส่วนค้นหา+หมวดหมู่+หน้ารายละเอียดสินค้าที่ยังไม่เคยมีสเปกมาก่อน เพื่อให้ระบบครบตามขอบเขตขั้นต่ำของ e-Commerce ทั้ง 9 ข้อ (ดูภาพขอบเขตที่ผู้ใช้ส่งมา): สมาชิก, จัดการสินค้า, ค้นหา+รายละเอียดสินค้า, ตะกร้า, สั่งซื้อ, ชำระเงินจำลอง, ติดตามสถานะ, แอดมินจัดการสินค้า/คำสั่งซื้อ, แดชบอร์ด

## เป้าหมาย

1. ลูกค้าค้นหาสินค้าด้วยคำค้นหาและกรองตามหมวดหมู่ได้
2. ลูกค้าเปิดหน้ารายละเอียดสินค้าแยกต่างหากได้ (`/product/:id`)
3. ลูกค้ากดยืนยันสั่งซื้อจากตะกร้า แล้วระบบบันทึกคำสั่งซื้อจริงลงฐานข้อมูล (สถานะเริ่มต้น `pending`)
4. หลังยืนยัน ระบบแสดงหน้า "ใบเสร็จ/สรุปคำสั่งซื้อ" พร้อม QR พร้อมเพย์แบบตัวอย่าง (ไม่ใช่เกตเวย์จริง)
5. ลูกค้าดูประวัติคำสั่งซื้อของตัวเอง พร้อมสถานะแต่ละรายการ จากข้อมูลจริง
6. แอดมินดูรายการคำสั่งซื้อทั้งหมด และเปลี่ยนสถานะได้ (pending → paid → shipped, หรือ cancelled)
7. แอดมินตั้งหมวดหมู่สินค้าตอนเพิ่ม/แก้ไขสินค้า
8. อัปเดตเอกสารโปรเจกต์ให้ตรงกับสถานะฟีเจอร์ใหม่

## ขอบเขตที่ตัดออกอย่างชัดเจน (Out of scope)

- ไม่มีการเชื่อมต่อผู้ให้บริการชำระเงินจริง (เช่น EasySlip) และไม่มีการตรวจสลิปอัตโนมัติ — ยังเป็นแผนงานรอบถัดไปตามที่ระบุใน analysis-and-design.md
- QR พร้อมเพย์ที่แสดงเป็นภาพ/ข้อความตัวอย่างเท่านั้น ไม่ใช่ QR ที่สร้างจากยอดเงินจริงผ่านมาตรฐาน EMVCo
- ไม่สร้างไฟล์ PDF จริง (ใช้หน้าเว็บสรุปคำสั่งซื้อที่พิมพ์ผ่าน browser print ได้แทน)
- ค้นหาสินค้าเป็นการกรองฝั่ง client จากรายการที่โหลดมาแล้วเท่านั้น ไม่มี full-text search ฝั่งฐานข้อมูล ไม่มี pagination (เหมาะกับจำนวนสินค้าระดับร้านค้าเล็ก)
- หมวดหมู่สินค้าเป็นชุดคงที่ 4 ค่าที่กำหนดในโค้ด ไม่มีหน้าจอจัดการหมวดหมู่แยกต่างหาก
- ไม่แก้ปัญหาอื่นที่ระบุไว้ใน analysis-and-design.md (เช่น ProfileEditModal, ตาราง profiles_admin, คอลัมน์ password) — อยู่นอกขอบเขตงานรอบนี้

## โครงสร้างฐานข้อมูล

### Migration 1: เพิ่มคอลัมน์ category ในตาราง products

```sql
alter table products
  add column category text not null default 'general'
  check (category in ('gaming', 'office', 'general', 'creator'));
```

ค่าเริ่มต้น `'general'` ทำให้สินค้าที่มีอยู่แล้วไม่พังจาก NOT NULL constraint สินค้าที่แอดมินเพิ่ม/แก้ไขต่อจากนี้ต้องเลือกหมวดหมู่จาก dropdown

หมวดหมู่ที่รองรับ (ค่า/label ที่แสดง):

| ค่าในฐานข้อมูล | label ที่แสดงในหน้าเว็บ |
| --- | --- |
| gaming | เกมมิ่ง |
| office | ทำงานออฟฟิศ |
| general | เรียน-ทำงานทั่วไป |
| creator | สตรีมมิ่ง/สร้างคอนเทนต์ |

### Migration 2: policy UPDATE บนตาราง orders (สำหรับแอดมิน)

ตรวจสอบ policy จริงในโปรเจกต์ Supabase (`BM_com_v2`, project id `ntcxxxstddjraothilqk`) แล้วพบว่าปัจจุบันมีแค่ policy `INSERT`/`SELECT` บน `orders` และ `order_items` เท่านั้น **ไม่มี policy `UPDATE` บนตาราง `orders` เลย** ดังนั้นแอดมินจะเปลี่ยนสถานะคำสั่งซื้อไม่ได้จนกว่าจะเพิ่ม migration ใหม่:

```sql
create policy "orders_update_admin"
  on orders for update
  to authenticated
  using (is_admin())
  with check (is_admin());
```

ทั้งสอง migration ต้อง apply ผ่าน `apply_migration` ของ Supabase MCP หรือ Supabase CLI ไม่ใช่แก้ผ่านหน้าเว็บ Dashboard ตรง ๆ เพื่อให้มีประวัติ migration ในโปรเจกต์ ต้อง apply ก่อนเริ่มเขียนโค้ดฝั่งหน้าเว็บที่พึ่งพา schema ใหม่

## Data flow

```
Home.jsx
  → useProducts() โหลดสินค้าทั้งหมด (เหมือนเดิม)
  → ช่องค้นหา (input text) + dropdown หมวดหมู่ → กรอง products ใน memory ด้วย useMemo
    เงื่อนไข: (name หรือ description มีคำค้นหาแบบ case-insensitive) AND (หมวดหมู่ตรงกับที่เลือก หรือเลือก "ทั้งหมด")
  → การ์ดสินค้าคลิกได้ (คลิกที่รูป/ชื่อ นำไป /product/:id, ปุ่มหยิบใส่ตะกร้ายังคงทำงานแยกโดยไม่นำทาง)

ProductDetail.jsx (/product/:id, ใหม่)
  → select products by id (public read, ไม่ต้อง login)
  → ถ้าไม่พบ แสดง "ไม่พบสินค้านี้"
  → แสดงรูปใหญ่ ชื่อ คำอธิบายเต็ม ราคา badge หมวดหมู่ ปุ่มหยิบใส่ตะกร้า

Cart.jsx
  → กด "ยืนยันคำสั่งซื้อ"
  → ตรวจสอบ session (ถ้าไม่ได้ล็อกอิน แสดงข้อความให้เข้าสู่ระบบ ไม่ทำต่อ)
  → insert orders (user_id, total_amount, status='pending') → ได้ order.id
  → insert order_items (order_id, product_id, quantity, price_at_time) ทีละรายการจาก cartItems
  → clearCart()
  → navigate('/order/:id')

OrderDetail.jsx (/order/:id, ใหม่)
  → select orders by id (+ order_items join products) — RLS จำกัดให้เห็นเฉพาะของตัวเองหรือแอดมิน
  → แสดงเลขคำสั่งซื้อ วันที่ รายการสินค้า ยอดรวม สถานะ และ QR ตัวอย่าง

History.jsx
  → ตรวจสอบ session (ถ้าไม่ได้ล็อกอิน แสดงข้อความให้เข้าสู่ระบบ)
  → select orders where user_id = ตัวเอง, order by created_at desc (+ นับจำนวนชิ้นต่อออเดอร์)
  → แต่ละแถวคลิกไปที่ /order/:id

Admin.jsx (ส่วนใหม่: จัดการคำสั่งซื้อ + หมวดหมู่ในฟอร์มสินค้า)
  → ฟอร์มเพิ่ม/แก้ไขสินค้าเดิม เพิ่ม dropdown เลือกหมวดหมู่ (บังคับเลือก, default 'general')
  → select orders ทั้งหมด (+ ชื่อ/อีเมลผู้ซื้อจาก profiles) order by created_at desc
  → เปลี่ยนสถานะ: update orders set status=... where id=...
```

## รายละเอียดแต่ละส่วน

### 1. Home.jsx — ค้นหา + กรองหมวดหมู่

- เพิ่มแถบเครื่องมือเหนือ grid สินค้า: ช่อง input ค้นหา (placeholder "ค้นหาสินค้า...") และ dropdown หมวดหมู่ (ตัวเลือกแรกคือ "ทุกหมวดหมู่")
- state `searchTerm` และ `selectedCategory` ใน component, กรองด้วย `useMemo` จาก `products` ที่ได้จาก `useProducts()` เพื่อไม่ต้อง query ซ้ำ
- การ์ดสินค้า: ครอบด้วย `Link to={`/product/${product.id}`}` เฉพาะส่วนรูป+ชื่อ+คำอธิบาย ส่วนปุ่ม "หยิบใส่ตะกร้า" อยู่นอก Link (หรือใช้ `stopPropagation` ถ้าจำเป็นต้อง nest) เพื่อไม่ให้คลิกปุ่มแล้วนำทางไปหน้ารายละเอียดโดยไม่ตั้งใจ
- ถ้ากรองแล้วไม่มีผลลัพธ์ แสดงข้อความ "ไม่พบสินค้าที่ค้นหา" แทน grid ว่าง

### 2. ProductDetail.jsx (ใหม่) — เส้นทาง `/product/:id`

- ใช้ `useParams()` ดึง product id จาก URL, โหลดด้วย `supabase.from('products').select('*').eq('id', id).single()`
- ถ้า error หรือไม่พบแถว แสดงข้อความ "ไม่พบสินค้านี้" พร้อมลิงก์กลับหน้าแรก
- แสดง: รูปใหญ่ (`image_url`), ชื่อ, badge หมวดหมู่ (แปลงค่าเป็น label ภาษาไทยตามตารางด้านบน), คำอธิบายเต็ม, ราคา, ปุ่ม "หยิบใส่ตะกร้า" (เรียก `addToCart` จาก `CartContext` เหมือน Home.jsx)

### 3. Cart.jsx — ปุ่มยืนยันคำสั่งซื้อ

- เพิ่มปุ่ม "ยืนยันคำสั่งซื้อ" ข้าง ๆ ปุ่ม "ล้างตะกร้า" เดิม แสดงเฉพาะเมื่อ `cartItems.length > 0`
- ต้องตรวจสอบ session ก่อน (ใช้ `supabase.auth.getSession()`) — ถ้าไม่มี session แสดงข้อความ "กรุณาเข้าสู่ระบบก่อนสั่งซื้อ" แทนที่จะเปิด modal ใหม่ (ปุ่มเข้าสู่ระบบมีอยู่แล้วที่ header) เพื่อไม่ต้องแก้โครงสร้าง modal state ที่อยู่ใน `MainLayout.jsx`
- ระหว่างบันทึก แสดงสถานะ loading และปิดปุ่มไม่ให้กดซ้ำ (ป้องกันสร้างออเดอร์ซ้ำจากการกดปุ่มรัว ๆ)
- ลำดับการเขียนข้อมูล: insert `orders` → ได้ id → insert `order_items` เป็น array เดียว (ไม่ loop เรียก insert ทีละแถว) → ถ้า insert order_items ล้มเหลว ให้แจ้ง error กับผู้ใช้ (ไม่ rollback order อัตโนมัติ เพราะเป็น client-side ธรรมดา ไม่มี transaction; ระบุ known limitation นี้ไว้ในเอกสาร)
- state `showQr` ที่มีอยู่เดิมใน Cart.jsx (ประกาศไว้แต่ไม่ได้ใช้งาน) ให้ลบออก เพราะ flow ใหม่นำทางไปหน้า OrderDetail แทนการแสดง QR ในหน้าเดิม

### 4. OrderDetail.jsx (ใหม่) — เส้นทาง `/order/:id`

- ใช้ `useParams()` ดึง order id จาก URL
- โหลดข้อมูล order พร้อม order_items และชื่อสินค้า (join ผ่าน `select` ของ supabase: `order_items(*, products(name, image_url))`)
- ถ้าไม่พบ order หรือไม่มีสิทธิ์เข้าถึง (RLS จะคืนค่าว่าง ไม่ error) ให้แสดงข้อความ "ไม่พบคำสั่งซื้อนี้"
- แสดง: เลขที่คำสั่งซื้อ (id), วันที่สั่งซื้อ, ตารางรายการสินค้า (ชื่อ, จำนวน, ราคาต่อชิ้นที่ price_at_time, ราคารวม), ยอดรวมทั้งหมด, badge สถานะ (สี/ข้อความตาม pending/paid/shipped/cancelled)
- แสดงกล่อง "ชำระเงินผ่าน PromptPay" เป็นภาพ/placeholder พร้อมข้อความอธิบายว่าเป็นตัวอย่าง และให้รอแอดมินตรวจสอบยอดโอนแล้วเปลี่ยนสถานะให้ — แสดงเฉพาะเมื่อสถานะเป็น `pending`
- มีปุ่ม "พิมพ์ใบเสร็จ" ที่เรียก `window.print()` (ใช้ print stylesheet พื้นฐานของ browser ไม่ต้องเขียน CSS พิเศษเพิ่ม)

### 5. History.jsx — ดึงข้อมูลจริง

- ตรวจสอบ session แบบเดียวกับ `Admin.jsx` (pattern `checkAdmin` แต่ไม่ต้องเช็ค role) — ถ้าไม่มี session แสดงข้อความให้เข้าสู่ระบบ (reuse layout เดียวกับ Admin's "สงวนสิทธิ์" card)
- query: `orders` ของ `user_id = session.user.id`, เรียงจากใหม่ไปเก่า, พร้อมนับจำนวน order_items ต่อออเดอร์ (`order_items(count)`)
- แสดงเป็นรายการ/การ์ด: วันที่, จำนวนชิ้นรวม, ยอดรวม, badge สถานะ, ลิงก์ไปหน้า `/order/:id`
- ถ้าไม่มีคำสั่งซื้อเลย แสดงข้อความเดิม "ยังไม่มีประวัติคำสั่งซื้อ"

### 6. Admin.jsx — หมวดหมู่ในฟอร์มสินค้า + ส่วนจัดการคำสั่งซื้อ

- ฟอร์มเพิ่ม/แก้ไขสินค้าที่มีอยู่แล้ว: เพิ่ม field dropdown "หมวดหมู่" (ตัวเลือกตามตารางด้านบน, ค่าเริ่มต้น `general`) ส่งค่า `category` ไปพร้อมกับ insert/update สินค้าเดิม
- เพิ่ม section ใหม่ต่อจาก "จัดการข้อมูลสินค้า" ชื่อ "จัดการคำสั่งซื้อ"
- โหลดรายการ orders ทั้งหมดพร้อมข้อมูลผู้ซื้อ (`profiles(full_name, email)`) เรียงใหม่ไปเก่า
- ตารางแสดง: วันที่, ชื่อ/อีเมลผู้ซื้อ, ยอดรวม, สถานะปัจจุบัน (badge), dropdown หรือปุ่มเปลี่ยนสถานะเป็น pending/paid/shipped/cancelled
- เปลี่ยนสถานะ: `supabase.from('orders').update({status}).eq('id', orderId)` แล้ว reload รายการ / อัปเดต state ทันที
- ปุ่มลิงก์ไปดูรายละเอียดออเดอร์ (`/order/:id`) เผื่อแอดมินอยากเห็นรายการสินค้าในออเดอร์นั้น

### 7. Route ใหม่ใน App.jsx

เพิ่มภายใต้ `MainLayout`:
- `<Route path="product/:id" element={<ProductDetail />} />`
- `<Route path="order/:id" element={<OrderDetail />} />`

## Error handling

- Insert order/order_items ล้มเหลว: แสดง alert แจ้งผู้ใช้ ไม่ clear ตะกร้า (ให้ลองกดใหม่ได้)
- โหลด order/product ไม่เจอ/ไม่มีสิทธิ์: แสดงข้อความแทน error technical
- เปลี่ยนสถานะฝั่งแอดมินล้มเหลว: แสดง alert เหมือน pattern เดิมใน Admin.jsx (`alert('...ไม่สำเร็จ')`)
- เพิ่ม/แก้ไขสินค้าโดยไม่เลือกหมวดหมู่: ป้องกันด้วย HTML `required` บน select และ default value เป็น 'general' อยู่แล้ว จึงไม่มีทางส่งค่าว่าง

## การทดสอบ

ทดสอบด้วยมือ (manual) ตาม pattern เดิมของโปรเจกต์นี้ (ดู [docs/test-report.md](../../test-report.md) และ [docs/uat.md](../../uat.md)):

1. พิมพ์คำค้นหาที่ตรงกับชื่อสินค้าบางส่วน → เห็นเฉพาะสินค้าที่ตรง, พิมพ์คำที่ไม่มีอยู่จริง → เห็นข้อความ "ไม่พบสินค้าที่ค้นหา"
2. เลือกหมวดหมู่จาก dropdown → เห็นเฉพาะสินค้าหมวดนั้น, กลับมาเลือก "ทุกหมวดหมู่" → เห็นสินค้าทั้งหมดอีกครั้ง
3. คลิกการ์ดสินค้า → ไปหน้า `/product/:id` เห็นรายละเอียดครบ, กดหยิบใส่ตะกร้าจากหน้านี้ได้ → ตะกร้ามีสินค้าเพิ่ม
4. เปิด `/product/:id` ด้วย id ที่ไม่มีอยู่จริง → เห็นข้อความ "ไม่พบสินค้านี้"
5. แอดมินเพิ่ม/แก้ไขสินค้าโดยเลือกหมวดหมู่ → สินค้าไปปรากฏถูกหมวดหมู่เวลากรองที่หน้าแรก
6. ลูกค้าเพิ่มสินค้าลงตะกร้า กดยืนยันคำสั่งซื้อโดยไม่ล็อกอิน → ต้องเห็นข้อความให้เข้าสู่ระบบ ไม่มี order ถูกสร้าง
7. ล็อกอินแล้วยืนยันคำสั่งซื้อ → มี order ใหม่สถานะ pending, ตะกร้าว่างลง, ไปหน้าใบเสร็จถูกต้อง
8. เข้าหน้าประวัติคำสั่งซื้อ → เห็นออเดอร์ที่เพิ่งสร้าง สถานะ pending
9. แอดมินเข้าหน้าจัดการคำสั่งซื้อ เปลี่ยนสถานะเป็น paid → ลูกค้า refresh หน้าประวัติ/ใบเสร็จเห็นสถานะเปลี่ยนตาม
10. ผู้ใช้ B พยายามเปิด `/order/:id` ของผู้ใช้ A ทาง URL ตรง → ต้องไม่เห็นข้อมูล (RLS บล็อก)

## เอกสารที่ต้องอัปเดตหลัง implementation

- `docs/data-schema.md`: เพิ่มคอลัมน์ `category` ในตาราง products (พร้อมตารางค่าที่รองรับ), เพิ่ม policy UPDATE บน orders ในตารางสรุป RLS
- `docs/analysis-and-design.md`: อัปเดตสถานะ FR-6, FR-8 เป็น "ใช้งานได้แล้ว (แบบยืนยันด้วยมือ ยังไม่มีเกตเวย์จริง)", เพิ่ม FR ใหม่สำหรับค้นหา/กรองหมวดหมู่/หน้ารายละเอียดสินค้าเป็น "ใช้งานได้แล้ว", ย้ายข้อที่เกี่ยวข้องออกจาก "แผนงานที่แนะนำ" ไปเป็นงานที่ทำแล้ว
- `docs/api.md`: เพิ่มตาราง endpoint สำหรับ `POST /rest/v1/orders`, `POST /rest/v1/order_items`, `GET /rest/v1/orders` (ของตัวเอง/แอดมินดูทั้งหมด), `PATCH /rest/v1/orders` (แอดมินเปลี่ยนสถานะ), `GET /rest/v1/products?id=eq...` (รายละเอียดสินค้า)
- `docs/how-it-works.md`: อธิบาย flow การค้นหา, ดูรายละเอียดสินค้า และการสั่งซื้อใหม่แบบภาษาคน
- `README.md`: ย้ายฟีเจอร์ "ค้นหา/กรองหมวดหมู่", "หน้ารายละเอียดสินค้า", "ยืนยันคำสั่งซื้อ" และ "ดูประวัติ/สถานะ" จากส่วน "อยู่ระหว่างวางแผน" มาเป็น "ใช้งานได้จริงตอนนี้" (คงข้อความเรื่อง QR พร้อมเพย์จริง/ตรวจสลิปอัตโนมัติไว้ในส่วนที่ยังไม่ได้พัฒนา)
