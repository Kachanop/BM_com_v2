# ออกแบบงาน: แก้บั๊ก CRUD สินค้าในแอดมิน + ชุดเอกสารโปรเจกต์ใหม่

วันที่: 2026-07-22

## บริบทและปัญหา

หลังสำรวจโค้ดและฐานข้อมูล Supabase (โปรเจกต์ `BM_com_v2`, id `ntcxxxstddjraothilqk`) พบว่า

1. ตาราง `products`, `orders`, `profiles` เปิด Row Level Security (RLS) ไว้ แต่ไม่มีกฎ (policy) ใดๆ เลย ทำให้ทุก role (anon และ authenticated) อ่าน/เขียนข้อมูลไม่ได้เลย แม้ในตาราง `products` จะมีข้อมูลจริงอยู่ 3 แถว นี่คือสาเหตุที่หน้าแรกและหน้าแอดมิน "สินค้าหาย" และแดชบอร์ดแอดมินโชว์ยอดขาย/จำนวนสมาชิก/จำนวนออเดอร์เป็น 0 ตลอด
2. หน้าแอดมิน (`frontend/src/pages/Admin.jsx`) รองรับแค่ดูสินค้า (Read) กับแก้ไขสินค้า (Update) เท่านั้น ยังไม่มีปุ่มเพิ่มสินค้าใหม่ (Create) และลบสินค้า (Delete)
3. หน้าเว็บคุยกับ Supabase โดยตรงผ่าน `@supabase/supabase-js` ไม่ได้ผ่าน backend Cloudflare Worker (`backend/src/index.ts`) เลย Worker ตอนนี้เป็น mock data ที่ไม่ถูกเรียกใช้จริงในแอป
4. ระบบตะกร้า/เช็คเอาต์/ชำระเงิน QR/ประวัติคำสั่งซื้อ ยังไม่ถูกพัฒนาจริง (หน้า Cart ไม่มีปุ่มสั่งซื้อ, หน้า History เป็นข้อความ static)

## ขอบเขตงานที่ตกลงกันไว้ (ยืนยันกับผู้ใช้แล้ว)

- ทำทั้งสองส่วนเรียงลำดับ: แก้บั๊ก/เพิ่มฟีเจอร์ก่อน แล้วค่อยเขียนเอกสาร
- CRUD ให้ครบเฉพาะ "สินค้า" เท่านั้น (ไม่รวมจัดการสมาชิก/สถานะคำสั่งซื้อในรอบนี้)
- แก้ RLS ให้ครบทั้ง 3 ตาราง (products, orders, profiles) เพราะเป็นบั๊กเดียวกัน และ orders/profiles จำเป็นต่อการทำงานของแดชบอร์ดแอดมิน
- คงสถาปัตยกรรมเดิม (หน้าเว็บคุยกับ Supabase ตรงๆ) ไม่ย้ายไปผ่าน Worker
- เอกสาร: README.md สรุปสั้น เป็นภาพรวม แล้วแยกไฟล์รายละเอียดไว้ใน `docs/`

## ส่วนที่ 1: แก้บั๊กฐานข้อมูล + เพิ่มฟีเจอร์ CRUD

### 1.1 Migration ฐานข้อมูล (Supabase)

สร้างฟังก์ชันช่วยตรวจสอบสิทธิ์แอดมินแบบ `SECURITY DEFINER` เพื่อไม่ให้เกิดปัญหา policy เรียกตัวเองวนซ้ำ (recursive RLS) เวลาเช็คว่า "คนนี้เป็นแอดมินไหม" จากตาราง `profiles`:

```sql
create or replace function public.is_admin()
returns boolean
language sql
security definer
set search_path = public
stable
as $$
  select exists (
    select 1 from public.profiles
    where id = auth.uid() and role = 'admin'
  );
$$;
```

จากนั้นเพิ่ม policy:

**products**
- SELECT: อนุญาตทุกคน (`to public using (true)`) เพราะเป็นแคตตาล็อกสินค้าที่ต้องเปิดให้ลูกค้าดูได้แม้ไม่ได้ล็อกอิน
- INSERT / UPDATE / DELETE: เฉพาะ `is_admin()`

**profiles**
- SELECT: เจ้าของแถว (`id = auth.uid()`) หรือ `is_admin()`
- UPDATE: เฉพาะเจ้าของแถว (`id = auth.uid()`)
- (ไม่ต้องมี INSERT policy เพราะการสร้างแถวใหม่ทำผ่าน trigger `handle_new_user` ที่เป็น SECURITY DEFINER อยู่แล้ว จึงข้าม RLS โดยธรรมชาติ)

**orders**
- SELECT: เจ้าของออเดอร์ (`user_id = auth.uid()`) หรือ `is_admin()`
- INSERT: ผู้ใช้ที่ล็อกอินแล้วสร้างออเดอร์ของตัวเองเท่านั้น (`user_id = auth.uid()`)

**order_items** (ยังไม่มี policy เลยแม้เปิด RLS ไว้ พบระหว่างสำรวจ ถือเป็นบั๊กเดียวกัน แก้พร้อมกัน)
- SELECT/INSERT: อนุญาตถ้า order_items แถวนั้นผูกกับ order ที่ผู้ใช้เป็นเจ้าของ หรือ `is_admin()`

Migration นี้จะใช้ Supabase MCP tool `apply_migration` กับโปรเจกต์ `ntcxxxstddjraothilqk` โดยตรง (เป็น remote project ที่ใช้งานจริงอยู่แล้ว ไม่มี local dev stack ในโปรเจกต์นี้)

### 1.2 ฝั่งหน้าบ้าน (Frontend)

**`frontend/src/hooks/useProducts.js`**: เพิ่มฟังก์ชัน
- `addProduct({ name, price, description, image_url })` → insert แถวใหม่ แล้ว reload
- `deleteProduct(id)` → delete แถวตาม id แล้ว reload
- ลบ `console.log` debug ที่ค้างอยู่จากการไล่บั๊กก่อนหน้า (เพราะ RLS แก้แล้ว ไม่จำเป็นต้องมี log เหล่านี้ในโค้ด production)

**`frontend/src/pages/Admin.jsx`**:
- เพิ่มช่องกรอก "รายละเอียดสินค้า" (description) ในฟอร์มแก้ไข/เพิ่มสินค้า เพราะหน้าแรกใช้แสดงคำอธิบายสินค้าอยู่แล้วแต่แอดมินยังแก้ไม่ได้
- เพิ่มปุ่ม "เพิ่มสินค้าใหม่" ที่เคลียร์ฟอร์มให้กรอกข้อมูลสินค้าใหม่ แล้วกดบันทึกเพื่อ insert
- เพิ่มปุ่ม "ลบสินค้า" ต่อรายการสินค้าที่เลือกอยู่ พร้อม `confirm()` ป้องกันลบพลาด
- ปรับ layout เดิมให้รองรับโหมด "กำลังเพิ่มสินค้าใหม่" กับ "กำลังแก้ไขสินค้าเดิม" (เช่น เปลี่ยนข้อความปุ่มบันทึก และล้าง selectedProductId เมื่ออยู่โหมดเพิ่ม)
- ตรวจสอบแล้วพบว่า `frontend/src/data/products.js` (ระบบเก็บสินค้าจำลองแบบ localStorage ยุคก่อนย้ายมาใช้ Supabase) ถูกเรียกใช้จากจุดเดียวคือปุ่ม "คืนค่าเริ่มต้น" ใน `Admin.jsx` และปุ่มนี้รีเซ็ตกลับไปเป็นข้อมูลจำลองในเครื่อง ไม่ได้แตะฐานข้อมูลจริงเลย ทำให้ผู้ใช้เข้าใจผิดได้ว่าข้อมูลถูกรีเซ็ตจริง จึงตัดสินใจ **ลบปุ่ม "คืนค่าเริ่มต้น" และไฟล์ `frontend/src/data/products.js` ทิ้งทั้งหมด** เพราะเป็นโค้ดที่ไม่ได้ใช้งานจริงและขัดแย้งกับสถาปัตยกรรมปัจจุบันที่ใช้ Supabase เป็นแหล่งข้อมูลเดียว

### 1.3 การทดสอบ

- รันแอป dev (`npm run dev` ใน `frontend/`) แล้วเปิดเบราว์เซอร์ยืนยันว่า
  - หน้าแรกแสดงสินค้าจาก Supabase ได้ (ทดสอบตอนไม่ได้ล็อกอิน)
  - ล็อกอินเป็นแอดมิน แล้วเพิ่ม/แก้/ลบสินค้าได้จริง และแดชบอร์ดตัวเลขไม่เป็น 0 ตลอดแล้ว (ตราบใดที่มีข้อมูล orders/profiles จริง)
  - ล็อกอินเป็นลูกค้าทั่วไปแล้วเข้าหน้าแอดมินไม่ได้ (ยังคง block ตามเดิม)

## ส่วนที่ 2: ชุดเอกสารโปรเจกต์ใหม่ (ภาษาไทย อ่านง่าย ไม่ใช้ em dash)

โครงสร้างไฟล์:

```
README.md                        (สรุปสั้น ภาพรวมโปรเจกต์ ลิงก์ไปเอกสารย่อยใน docs/)
docs/
  tech-stack.md                  (เทคโนโลยีที่ใช้ และหน้าที่ของแต่ละส่วน)
  how-it-works.md                (อธิบายการทำงานของระบบแบบภาษาคน ไม่เน้นศัพท์เทคนิค)
  use-case-and-sequence.md       (use case diagram + sequence diagram หลักๆ ด้วย mermaid)
  data-schema.md                 (ER diagram ด้วย mermaid + อธิบายแต่ละตาราง/คอลัมน์)
  api.md                         (เอกสาร API: Supabase REST ที่ใช้งานจริง + Worker mock endpoints ที่ยังไม่ได้ใช้งานจริง ระบุชัดเจนว่าอันไหน current อันไหน prototype)
  analysis-and-design.md         (การวิเคราะห์ความต้องการ, ผู้ใช้งานระบบ, การออกแบบเชิงสถาปัตยกรรม)
  sdlc.md                        (วงจรการพัฒนาโปรเจกต์นี้ พร้อม mermaid diagram ของขั้นตอน)
  swagger.yaml                   (สเปก OpenAPI 3.0 ของ REST endpoint ที่มีจริง สำหรับเปิดดูใน Swagger UI/Editor)
```

Diagram ที่จะทำด้วย mermaid (ฝังในไฟล์ .md โดยตรง เพื่อให้ GitHub render ให้อัตโนมัติ):
- Use case diagram (ลูกค้า, แอดมิน)
- Sequence diagram: ลูกค้าดูสินค้าหน้าแรก, แอดมินเพิ่ม/แก้/ลบสินค้า, การสมัครสมาชิก/ล็อกอิน, การตรวจสิทธิ์แอดมิน
- ER diagram: products, orders, order_items, profiles
- Flowchart: SDLC ของโปรเจกต์นี้

ทุกไฟล์เขียนด้วยภาษาไทยล้วน หลีกเลี่ยงศัพท์เทคนิคที่ไม่จำเป็น ใช้คำเชื่อมประโยคปกติแทนเครื่องหมาย em dash (—) และเน้นให้คนที่ไม่มีพื้นฐานเขียนโปรแกรมอ่านเข้าใจได้

เอกสารจะระบุชัดเจนว่าฟีเจอร์ไหน "ใช้งานได้แล้วจริง" กับฟีเจอร์ไหนที่ README เดิมพูดถึงแต่ยังไม่ได้พัฒนาจริง (เช่น QR PromptPay, EasySlip, ประวัติคำสั่งซื้อ) เพื่อความถูกต้องตรงไปตรงมา ไม่หลอกผู้อ่าน

## ความเสี่ยงและสิ่งที่ต้องระวัง

- การรัน migration บนโปรเจกต์ Supabase จริง (ACTIVE_HEALTHY, มีข้อมูลจริงอยู่แล้ว) ต้องเป็น `alter`/`create policy` ที่ไม่ลบข้อมูลเดิม ปลอดภัยต่อข้อมูลที่มีอยู่
- ต้องตรวจสอบว่า `frontend/src/data/products.js` ยังถูกใช้ที่อื่นไหมก่อนลบส่วนที่เกี่ยวข้องออกจาก Admin.jsx
- เอกสาร Swagger จะอ้างอิงตาม REST surface ที่ Supabase auto-generate ให้ตาราง `products` (PostgREST) ซึ่งต้องระบุ URL/anon key แบบ placeholder ไม่ใส่ค่าจริงที่หลุดเป็นความลับ
